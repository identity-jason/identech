The Perils of Loading Assemblies in powershell

After adding the script to load a new DLL into the GAC and AIC, I've been doing some additional work on my workflow libraries - including moving some functions into a central library rather than copying and pasting code between workflow activities because that always ends well...

So when I added a new class to my Workflow Activity I was a little bemused when it didn't become available when I enumerated the public methods of the containing DLL via the GetExportedTypes() method - it was particularly odd (with hindsight) as a previous DLL I'd altered was fine, but the one I'd amended - by copying and pasting in the code from the previous build didn't appear to have the new methods available.

Cue the inevitable "hokey-cokey" classes where I'd add the new class, rebuild, test via powershell, remove it and try again (now that was a fun afternoon!) before spotting that if I quit powershell and reloaded it then the "missing" class suddenly became visible.

Turns out that if I'd been paying attention, I'd have realized that .net doesn't allow unloading of assemblies - and while with a standalone executable it's "AppDomain" is destroyed at the end of execution, effectively unloading anything you've loaded, powershell counts it's session rather than the lifetime of a script as it's execution and once you've loaded the DLL in once, it's there for the lifetime of the powershell instance.

Seem the way to get around it would be either to try stand something up via a temporary App Domain, but quite how to get that into Powershell may be a whole new adventure - most of the people talking about it online are creating wrapper applications around their dll loads to artificially partition up and protect their powershell sessions - so in a nutshell, if you're seeing odd behaviour with the load script OR are getting lots of dll open in another application that you're not expecting, it may well be worthwhile quitting powershell and opening a new session to see if that fixes things.

"Switch it off and back on again" still rules - even with the joys of Powershell and .net!
