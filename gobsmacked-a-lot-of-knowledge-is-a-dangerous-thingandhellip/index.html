Identity Technology (again)
August 2, 2011

Gobsmacked (a lot of knowledge is a dangerous thing&hellip;)

I’ve recently being working with some colleagues in Germany and was asked to take a look at installing FIM onto a server – as ever, there’s an incredible mountain of things to do, but I figured I could spare the 15 minutes to run the install while I was waiting for some code to compile and no harm would be done.

At the first install, everything worked fine until I got to the service account page, filled in the blanks and….

This is where the nightmare began – the server returned with:

The service account cannot access SQL server. Ensure that the server is accessible, the service account is not a local account being used with a remote SQL Server, and that the account doesn’t already have a SQL login

So, I checked everything – firewalls, domains, credentials, SQL permissions and verified the connection from powershell to no avail. Ok, it’s SQL so I defined alias to no avail, installed patches, upgraded clients everything that I and the team I’m working with could think to do, we did – in spades.

Finally, I got to the point of desperation and tried what couldn’t possibly be the solution based on a blog entry I found on the exceedingly well named “Over 5 minutes – answers for questions that should be simple”, and changed the case of the domain – instant success, go directly to the group specification page and be on your way.

So that’s it – if I’d known less about the complicated ways to install FIM, maybe I’d have got to this point sooner – or maybe I’ll learn to check the simple things first next time (now that I know to check
